서버 동기화 및 공유 설계

현재 GitHub Pages 프론트엔드는 유지하면서 생활비 고정비 관리자에 백엔드를 추가합니다.

목표

첫 번째 서버 기능은 계정 기반 동기화와 백업입니다. 그 다음 단계로 가족 또는 생활비 공동 관리용 워크스페이스 공유를 추가합니다. 일회성 실제 지출 기록과 완전한 가계부 기능은 이 설계 범위에서 제외합니다.

우선순위

  1. 1계정 기반 동기화와 백업
  2. 2가족 또는 공동 생활비 관리를 위한 워크스페이스 공유
  3. 3실제 지출 기록은 보류

아키텍처

모노레포 구조를 사용합니다.

living-cost-manager/
  apps/
    web/
    api/
  packages/
    shared/
  prisma/
    schema.prisma
  • apps/web: 기존 Next 정적 프론트엔드입니다. GitHub Pages에 배포합니다.
  • apps/api: Fastify API 서버입니다. OCI VM에 배포합니다.
  • packages/shared: Zod 스키마, DTO 타입, enum, API 계약을 공유합니다.
  • prisma/schema.prisma: PostgreSQL 데이터베이스 모델입니다.

프론트엔드는 Prisma 모델을 직접 가져오지 않습니다. 프론트와 백엔드는 packages/shared의 DTO와 Zod 스키마만 공유합니다.

기술 선택

백엔드

  • Fastify
  • TypeScript
  • Zod 기반 요청/응답 검증
  • JWT 인증
  • Prisma ORM
  • PostgreSQL
  • OCI VM에서 Docker Compose로 운영

프론트엔드

  • 기존 Next static export 유지
  • GitHub Pages 호스팅 유지
  • 기존 localStorage 데이터는 오프라인 캐시와 마이그레이션 원본으로 유지

데이터 모델

핵심 테이블

  • User
  • Workspace
  • WorkspaceMember
  • Category
  • PaymentCard
  • FixedCost
  • BackupSnapshot

1차 릴리스 동작

  • 사용자는 회원가입 또는 로그인을 합니다.
  • 사용자마다 개인 워크스페이스가 자동 생성됩니다.
  • 카테고리, 카드, 고정비 항목은 해당 워크스페이스 아래에 저장됩니다.
  • 앱은 서버와 현재 브라우저 사이에서 워크스페이스 예산 데이터를 동기화할 수 있습니다.

이후 공유 기능 동작

  • 워크스페이스 owner는 멤버를 초대할 수 있습니다.
  • 멤버는 같은 워크스페이스에 접근할 수 있습니다.
  • 역할은 owner, editor, viewer입니다.
  • owner는 멤버와 모든 예산 데이터를 관리할 수 있습니다.
  • editor는 예산 데이터를 수정할 수 있지만 멤버는 관리할 수 없습니다.
  • viewer는 예산 데이터를 조회만 할 수 있습니다.

공유 API 계약

packages/shared는 다음 항목을 정의합니다.

  • PaymentMethodId
  • WorkspaceRole
  • UserDto
  • WorkspaceDto
  • WorkspaceMemberDto
  • CategoryDto
  • PaymentCardDto
  • FixedCostDto
  • 카테고리, 카드, 고정비 생성/수정 스키마
  • 동기화 요청/응답 스키마

FixedCostDto는 현재 클라이언트 모델을 유지합니다.

  • id
  • workspaceId
  • name
  • categoryId
  • paymentMethodId
  • paymentOptionId
  • amount
  • periodMonths
  • billingDay

periodMonths는 소수점 첫째 자리까지 반올림된 숫자로 유지합니다.

동기화 모델

localStorage는 캐시와 fallback으로 유지합니다.

로그인 시

  1. 브라우저의 로컬 데이터를 읽습니다.
  2. 서버의 워크스페이스 데이터를 읽습니다.
  3. 서버 워크스페이스가 비어 있으면 로컬 데이터를 업로드할지 묻습니다.
  4. 로컬 데이터와 서버 데이터가 모두 있고 내용이 다르면 사용자가 선택하게 합니다.
    • 서버 데이터 사용
    • 이 브라우저 데이터 업로드
    • 일단 로컬 전용으로 유지

동기화 활성화 후

  • 일반 편집은 먼저 로컬 상태에 즉시 반영합니다.
  • 앱은 변경사항을 API로 전송합니다.
  • API 호출이 실패하면 localStorage가 최신 브라우저 상태를 보존하고 UI에 동기화 경고를 표시합니다.
  • 수동 "지금 동기화" 동작으로 실패한 변경사항을 다시 전송합니다.

API 표면

인증

  • POST /auth/register
  • POST /auth/login
  • GET /me

워크스페이스

  • GET /workspaces
  • POST /workspaces
  • GET /workspaces/:workspaceId

동기화

  • GET /workspaces/:workspaceId/snapshot
  • PUT /workspaces/:workspaceId/snapshot

공유 기능: 개인 동기화 이후 구현

  • GET /workspaces/:workspaceId/members
  • POST /workspaces/:workspaceId/invitations
  • PATCH /workspaces/:workspaceId/members/:memberId
  • DELETE /workspaces/:workspaceId/members/:memberId

첫 구현은 세부 CRUD API를 많이 만들기보다 snapshot 동기화를 우선합니다. 이렇게 하면 현재 클라이언트 상태 구조와 잘 맞고 서버 마이그레이션 범위도 작아집니다.

보안

  • 비밀번호는 현대적인 password hashing 방식으로 해시합니다.
  • JWT secret은 절대 커밋하지 않습니다.
  • API CORS는 GitHub Pages origin만 허용합니다.
  • 모든 워크스페이스 route는 멤버십을 검증합니다.
  • 서버 mutation에는 역할 기반 권한 검사를 적용합니다.
  • 백업과 snapshot은 워크스페이스 단위로 격리합니다.

현재 프론트엔드에 허용할 CORS origin은 다음과 같습니다.

https://sanghyun-io.github.io

/living-cost-manager 경로는 CORS origin에 포함되지 않습니다.

배포

프론트엔드

  • GitHub Pages를 유지합니다.
  • 현재 /living-cost-manager base path를 유지합니다.
  • production build에 API base URL 설정을 추가합니다.

백엔드

  • apps/api를 Docker Compose로 OCI VM에 배포합니다.
  • 첫 서버 릴리스에서는 PostgreSQL도 Docker로 실행합니다.
  • 환경 변수는 repo가 아니라 VM에 저장합니다.
  • 인증 endpoint를 공개하기 전 reverse proxy를 통해 HTTPS를 적용합니다.

현재 앱에서의 마이그레이션

현재 앱은 localStorage 우선 구조입니다. 서버 도입 과정에서 로컬 데이터를 잃지 않아야 합니다.

  1. 사용자가 기존 Pages 앱을 엽니다.
  2. 새 서버 계정으로 로그인합니다.
  3. 앱이 브라우저 로컬 데이터를 감지합니다.
  4. 앱은 해당 로컬 데이터를 개인 워크스페이스에 업로드할지 묻습니다.
  5. 업로드 후에는 서버 snapshot을 source of truth로 사용합니다.

.lcm 전체 백업 export/import는 서버 동기화 여부와 관계없이 계속 제공합니다.

테스트

백엔드 테스트

  • 회원가입/로그인
  • 워크스페이스 멤버십 검증
  • snapshot 읽기/쓰기
  • 역할 기반 mutation 거부
  • 잘못된 고정비 주기 값 validation 거부

shared package 테스트

  • DTO 스키마가 현재 고정비/카테고리/카드 shape를 허용하는지 확인
  • 소수점 periodMonths가 첫째 자리까지 보존되는지 확인

프론트엔드 테스트

  • 로컬 전용 모드가 계속 동작하는지 확인
  • 로컬 데이터가 있을 때 로그인 동기화 안내가 뜨는지 확인
  • 서버 snapshot으로 대시보드를 hydrate할 수 있는지 확인
  • 동기화 실패 시 로컬 편집이 유지되고 경고가 표시되는지 확인

명시적 제외 범위

  • 일회성 실제 지출 기록
  • 은행/카드 거래내역 import
  • 영수증 OCR
  • 자동 납부 알림
  • 네이티브 모바일 앱
  • 프론트엔드를 GitHub Pages 밖으로 이전하는 작업
